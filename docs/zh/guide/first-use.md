# 首次使用

本文以一次受控测试为例，帮助你确认 V3il 的主要运营链路已经连通。

## 1. 查看 Command Center

登录后先查看 Command Center，确认 Managed Host、Sandbox Image、检测服务、智能体模型和 LightRAG 状态正常。

如果某项基础资源不可用，先在对应管理页面处理，再创建环境。

## 2. 设计测试场景

选择一个边界清楚的场景，例如内部管理站、文件交换服务或带登录流程的业务应用。提前写下三项内容：

- 希望呈现给攻击者的业务背景；
- 希望观察的行为与决策；
- 可以接受的网络暴露和外联范围。

这些信息将成为后续环境设计和调查复核的共同依据。

## 3. 创建欺骗环境

在 Deception Environments 中选择 Managed Host、Sandbox Image、外联策略和适配模式，并添加必要的参考资料。

进入环境 Agent Console 后，用自然语言说明服务结构、身份关系、数据内容、用户路径和观察目标。Ph4ntom 会提出环境设计并完成部署与验证。

## 4. 检查环境

在环境工作区确认：

- 服务和端口已经上线；
- 页面、接口、身份和数据符合场景；
- 环境版本记录了本次设计；
- 行为观测与检测状态正常。

从隔离测试网络访问环境，执行几项预期交互，例如登录尝试、目录访问、命令执行或文件操作。

## 5. 查看 Incident

当行为被关联为 ThreatIncident 后，进入 Incident 工作区查看：

- 涉及的环境和时间范围；
- 行为与检测时间线；
- 当前调查任务及负责人；
- 已记录的证据和分析；
- 智能体运行与审计记录。

如果行为没有进入预期 Incident，检查检测策略、来源关系和时间范围。

## 6. 观察智能体协作

关注 V3il 如何拆分调查问题，以及 H4wk、Ph4ntom、L1ly 和 J4ck 的任务分工。复查任务是否覆盖关键行为、证据是否支持分析、不同结论之间是否一致。

必要时可在 Incident Console 中补充背景、提出问题或调整调查优先级。

## 7. 测试动态诱导

根据当前调查，提出一个需要环境验证的问题，例如攻击者是否会继续访问某类数据或使用某条路径。

在 `manual_approval` 模式下复查 Ph4ntom 的调整目的、风险和预期效果后再批准。调整上线后，确认新行为继续进入原有调查上下文。

## 8. 形成报告

当关键任务与分析完成后，检查攻击意图、攻击链、指标、画像、风险和响应建议。推进 Incident 进入报告阶段，生成最终报告并导出证据包。

报告发布到 LightRAG 后，可以在 Knowledge Base 中检索本次调查内容。

## 9. 完成复盘

首次测试结束后，建议记录：

- 环境设计是否足够可信；
- 哪些信号对关联和调查最有价值；
- 智能体分工是否符合团队流程；
- 哪些环境调整有效；
- 报告是否满足内部交付标准；
- 网络隔离、数据留存和权限配置是否需要调整。
